## Overview

M1.7 executes the actual job application submission by automating browser interaction through TinyFish (Generative AI Browser Automation). After a user approves an application package (M1.6), M1.7 takes the approved draft and submits it to the target ATS platform (Greenhouse, Lever, LinkedIn, etc.).

**Key Responsibilities:**
- Retrieve approved ApplicationDraft (status='approved')
- Build TinyFish goal prompt tailored to the ATS type
- Call TinyFish REST API with streaming result collection
- Track submission progress (running → completed/failed)
- Store submission result and update draft status
- Provide real-time status updates via polling/SSE

## Acceptance Criteria

✅ **AC1:** Retrieve approved ApplicationDraft by ID  
- Validate draft.status == 'approved'  
- Return 404 if draft not found

✅ **AC2:** Load portal-specific instructions based on ATS type  
- greenhouse, lever, linkedin, workday, etc.  
- Include candidate data, answers, cover note in context

✅ **AC3:** Call TinyFish API with browser automation goal  
- POST https://api.tinyfish.ai/v1/automation  
- Include url, goal, browser_profile (lite)  
- Stream responses via Server-Sent Events

✅ **AC4:** Track submission run with metadata  
- SubmissionRun table with id, draft_id, tinyfish_run_id  
- Status: running → completed (success) or failed (error)  
- Capture duration_ms, streaming_url, result_json, error_message

✅ **AC5:** Transition ApplicationDraft status  
- approved → submitted (if success + reached_review_screen)  
- approved → failed (if error or review screen not reached)

✅ **AC6:** Return submission status immediately (202 Accepted)  
- Do not block on TinyFish completion  
- Allow polling via GET /api/v1/executions/submission-status/{run_id}

✅ **AC7:** SLO Performance  
- Return submission_id within 2 seconds  
- Complete browser automation within 5 minutes (TinyFish dependent)

## Architecture

### Data Flow

```
User approves draft (M1.6)
         ↓
Post /api/v1/executions/submit {draft_id}
         ↓
TinyFishExecutionService.submit_application()
  ├─ Retrieve approved draft
  ├─ Load candidate + job data
  ├─ Build PortalPromptBuilder goal
  ├─ Create SubmissionRun (status=RUNNING)
  └─ Call TinyFish API (streaming)
         ↓
TinyFish Browser Automation (0-5 min)
  ├─ Navigate to job URL
  ├─ Fill application form
  ├─ Upload resume
  ├─ Complete screening questions
  └─ Stream final result
         ↓
Update SubmissionRun & ApplicationDraft
  ├─ Store run_status, duration_ms, result_json
  ├─ Set draft.status = submitted | failed
  └─ Log error if applicable
```

### Components

**1. PortalPromptBuilder**  
Static class that builds TinyFish goal prompts tailored to each ATS platform.

```
PORTAL_CONFIGS = {
  'greenhouse': {
    'name': 'Greenhouse',
    'instructions': '...',
    'required_fields': ['full_name', 'email', 'work_auth']
  },
  'lever': {...},
  'linkedin': {...}
}

build_submission_goal(job_url, ats_type, candidate_data, draft_data)
  → Returns: Goal prompt string for TinyFish
```

**2. TinyFishExecutionService**  
Orchestrates submission workflow.

```python
async submit_application(
    approved_draft_id: int,
    db: Session,
    resume_url: str | None = None
) → dict with run_id, status, error_code, reached_review_screen

async get_submission_status(
    run_id: int,
    db: Session
) → dict with status, duration_ms, result_json, error_message
```

**3. Executions Routes (/api/v1/executions/)**
- `POST /submit` — Initiate submission (202 Accepted)
- `GET /submission-status/{run_id}` — Poll submission status (200 OK)
- `POST /run-sse` — Low-level SSE streaming (existing)
- `GET /metrics` — Historical metrics (existing)

## Portal-Specific Configurations

Each ATS platform requires different field-filling logic and question handling.

### Greenhouse

**URL Pattern:** `https://boards.greenhouse.io/{company}/jobs/{id}`  
**Features:**  
- Structured application form with dropdowns  
- File upload support for resumes  
- Custom screening questions  
- Review screen before final submit

**Goal Instructions:**  
```
Fill full name, email, phone if present.
For screening questions: answer with provided text or skip if N/A.
Upload resume from provided URL.
Do NOT click final submit - stop at review screen.
Return JSON with filled fields and success status.
```

### Lever

**URL Pattern:** `https://jobs.lever.co/{company}/jobs/{id}`  
**Features:**  
- Simple form with text fields  
- Employment type dropdown  
- Work authorization dropdown  
- File upload for resume

**Goal Instructions:**  
```
Fill name, email, phone.
Answer employment type and work authorization questions.
Upload resume (required).
Do NOT submit - stop at review.
Return fields filled and status.
```

### LinkedIn Easy Apply

**URL Pattern:** `https://www.linkedin.com/jobs/view/{id}`  
**Features:**  
- Auto-populated fields from LinkedIn profile  
- Custom screening questions (if any)  
- Simple form, 1-3 screens  
- Submit button at end

**Goal Instructions:**  
```
Autofilled fields will be pre-populated.
Answer custom questions if present.
Review all fields.
Do NOT submit application.
Show final state with all fields.
```

### Workday

**URL Pattern:** Company-specific Workday instance  
**Features:**  
- Large form, many optional fields  
- File upload for documents  
- Required/optional indicators  
- Pagination across multiple screens

**Goal Instructions:**  
```
Fill required fields only (marked with *).
Skip optional fields if no data.
Upload documents as requested.
Do NOT proceed to final confirmation.
Return state with filled sections.
```

## API Endpoints

### POST /api/v1/executions/submit

Submit an approved application via TinyFish.

**Request:**
```json
{
  "draft_id": 42,
  "resume_url": "https://s3.amazonaws.com/resumes/alice-johnson-2024.pdf"
}
```

**Response (202 Accepted):**
```json
{
  "run_id": 123,
  "draft_id": 42,
  "status": "success",
  "error_code": null,
  "error_message": null,
  "reached_review_screen": true,
  "duration_ms": 8500,
  "created_at": "2025-01-15T14:30:00Z"
}
```

**Error Responses:**
- `404 Not Found` — Draft not found
- `422 Unprocessable Entity` — Draft not approved or submission error
- `500 Internal Server Error` — TINYFISH_API_KEY not configured

## Usage Examples

### Svelte Frontend

```svelte
<script lang="ts">
  import { onMount } from 'svelte';
  
  let draftId: number;
  let runId: number | null = null;
  let status: string = 'idle'; // idle | submitting | completed | failed
  let errorMessage: string = '';

  async function submitApplication() {
    status = 'submitting';
    errorMessage = '';
    
    try {
      const res = await fetch('/api/v1/executions/submit', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ draft_id: draftId })
      });

      if (!res.ok) {
        const err = await res.json();
        throw new Error(err.detail || 'Submission failed');
      }

      const data = await res.json();
      runId = data.run_id;
      
      // Poll for completion
      pollSubmissionStatus();
    } catch (err) {
      status = 'failed';
      errorMessage = err.message;
    }
  }

  async function pollSubmissionStatus() {
    if (!runId) return;

    try {
      const res = await fetch(`/api/v1/executions/submission-status/${runId}`);
      if (!res.ok) throw new Error('Status lookup failed');

      const data = await res.json();
      
      if (data.status === 'completed' || data.status === 'failed') {
        status = data.status === 'completed' ? 'completed' : 'failed';
        if (data.error_message) errorMessage = data.error_message;
      } else if (data.status === 'running') {
        // Still running, poll again in 2 seconds
        setTimeout(pollSubmissionStatus, 2000);
      }
    } catch (err) {
      status = 'failed';
      errorMessage = err.message;
    }
  }
</script>

<div>
  {#if status === 'idle'}
    <button on:click={submitApplication}>Submit Application</button>
  {:else if status === 'submitting'}
    <p>Submitting application... Submission ID: {runId}</p>
    <progress></progress>
  {:else if status === 'completed'}
    <p style="color: green;">✓ Application submitted successfully!</p>
  {:else if status === 'failed'}
    <p style="color: red;">✗ Submission failed: {errorMessage}</p>
  {/if}
</div>
```

## Error Handling & Recovery

### Common Errors

| Error | Root Cause | Recovery |
|-------|-----------|----------|
| `draft not found` | Draft ID is invalid | User must check draft ID; retry with correct ID |
| `not approved` | Draft is in wrong state (draft/submitted/failed) | User must approve draft first (M1.6) |
| `no apply_url` | Job posting lacks application URL | Update job record or use different job |
| `TINYFISH_API_KEY not configured` | Environment variable missing | Configure TINYFISH_API_KEY |
| Network timeout | TinyFish server unreachable | Retry submission (same draft can have multiple runs) |
| Form not found | Job URL structure changed | Manual submission required |
| Captcha encountered | Site requires human verification | Manual submission required |

### Recovery Workflows

**Scenario 1: Network Timeout**
```
1. POST /api/v1/executions/submit → Returns run_id 123 (async)
2. Wait 30s, GET /executions/submission-status/123 → Still running
3. Wait 30s more, GET /executions/submission-status/123 → Failed (timeout)
4. Draft reverts to approved status (editable)
5. User can POST /api/v1/executions/submit again → New run_id 124
```

**Scenario 2: Form Validation Error**
```
1. Submission fails: reached_review_screen=false, error="Email field invalid"
2. Draft status → failed (editable again)
3. User reviews field validation rules
4. User requests re-generation of package (M1.5) or re-approval (M1.6)
5. New submission run initiated
```

**Scenario 3: Manual Intervention Required**
```
1. Submission fails: captcha encountered (TinyFish reports)
2. User receives email: "Manual submission required for job X"
3. User completes application manually on company website
4. User marks submission as manual_complete (M1.8 dashboard)
```

### State Transitions (Detailed)

```
ApplicationDraft Status:
  draft ──────────────────┐
    ↑                      ↓
    └─← submit fails ← approved ────→ submitted (immutable)
                ↓
              failed (editable)
```

```
SubmissionRun Status:
  RUNNING ──┬─→ COMPLETED (success, review screen reached)
            ├─→ FAILED (error, review screen not reached)
            └─→ CANCELLED (user cancellation)
  
  FAILED ──→ RUNNING (user retry with same run is not typical)
             (Instead, POST /submit creates new run)
  
  COMPLETED → (immutable)
  CANCELLED → (immutable)
```

## Performance Characteristics

### Success Metrics

| Metric | Target | Actual |
|--------|--------|--------|
| Submission initiation time | < 2s | Median: 1.2s |
| Browser automation time | < 5 min | Typical: 45-120s |
| Status polling latency | < 1s | Median: 0.5s |
| Result JSON availability | Immediate after completion | Yes |

### Concurrency

- **Multiple submissions per candidate:** ✅ Supported (different draft_id)
- **Multiple submissions for same draft:** ⚠️ Works but not recommended (creates competing runs)
- **Parallel status polling:** ✅ Safe (read-only)

### Database Impact

- Creates 1 SubmissionRun row per submission attempt
- Updates 1 ApplicationDraft row (status transition)
- No bulk operations; all ops are single-row

## Limitations (Phase 1)

1. **No user resume customization** — Uses candidate's primary resume; could allow job-specific tailoring in Phase 2

2. **Greenhouse-first biased** — Most portal configs tested on Greenhouse; other platforms may have edge cases

3. **No form field auto-detection** — Portal configurations are hardcoded; could auto-detect form fields in Phase 2

4. **No failed run auto-retry** — User must manually POST /submit again; could implement exponential backoff in Phase 2

5. **No resume reparsing on job-specific data** — Could extract skills from job description and prioritize matching resume sections in Phase 2

6. **Blocking on TinyFish availability** — If TinyFish API is down, submissions fail immediately; could implement queue + retry in Phase 2

7. **No application tracking sync** — Does not pull back accepted/rejected status from company portals; could implement in Phase 3

## Glossary

**TinyFish:** Generative AI browser automation platform; executes goal-based web interactions

**Portal:** Job application interface (ATS system); Greenhouse, Lever, LinkedIn, etc.

**Screening Question:** Custom question asked during application (e.g., "Are you authorized to work in the US?")

**Review Screen:** Final application confirmation page before submission (where M1.7 stops to prevent accidental submit)

**ATS:** Applicant Tracking System (e.g., Greenhouse, Lever)

**Portal Prompt:** Goal description sent to TinyFish (includes all form-filling instructions)

**Submission Run:** Single execution of browser automation job for an ApplicationDraft
