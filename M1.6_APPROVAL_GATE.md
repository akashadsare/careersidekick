# M1.6: Human Approval Gate

**Status:** ✅ Complete  
**Commit:** [Pending]

## Overview

M1.6 provides the hard approval screen before submission. This is the critical human review gate where:

1. **User Reviews Package** — Full draft shown with candidate name, job details, fit score
2. **User Can Edit** — Modify any answer or cover note before approval
3. **User Approves Explicitly** — Must click "Approve" button (no auto-submit)
4. **Snapshot Created** — Immutable copy stored in S3 for audit trail
5. **Draft Transitioned** — Status changes from 'draft' → 'approved' (ready for M1.7 submission)

**User Flow:**
```
Candidate generates package (M1.5)
        ↓
Views approval screen (GET /api/v1/approvals/{draft_id})
        ↓
Edits answers/cover note (PATCH /api/v1/approvals/{draft_id})
        ↓
Sees live preview with edits
        ↓
Clicks "Request Revisions" if needed (POST reject)
        ↓
OR Clicks "Approve & Continue" 
        ↓
Snapshot stored in S3
        ↓
Draft.status = 'approved'
        ↓
Ready for M1.7 submission
```

- ✅ Display full package on approval screen (answers, cover note, fit score)

## Implementation Details

### Database Schema

**ApplicationDraft** (status field extended):
```
status: enum (draft | approved | submitted | failed)
  - draft: Editable, not approved yet (M1.5, M1.6 pre-approval)
  - approved: Snapshot taken, ready for submission (M1.6 post-approval)
  - submitted: Application completed (M1.7+)
  - failed: Submission or execution failed
```

### Service Architecture

**File:** `backend/app/services/approval_service.py` (~180 lines)

**Class: ApprovalService**

#### Methods

**`get_draft_for_approval(draft_id, db) → dict`**
- Retrieve draft for approval screen
- Fetch candidate + job details
- Validate draft is in 'draft' status
- Return complete package with all answers

**`update_draft_answers(draft_id, updated_answers, updated_cover_note, db) → dict`**
- Update answer text (user edits)
- Update cover note
- Validate draft is in 'draft' status
- Persist changes to database
- Return updated draft

**`approve_draft(draft_id, db, s3_bucket) → dict`**
- Create snapshot JSON
- Store snapshot in S3 (if available)
- Transition draft.status from 'draft' → 'approved'
- Return approval confirmation

**`reject_draft(draft_id, reason, db) → dict`**
- Log rejection reason in draft
- Keep draft in 'draft' status (remains editable)
- Return updated draft for re-editing

### API Endpoints

**GET /api/v1/approvals/{draft_id}**

Retrieve draft for approval screen.

Response:
```json
{
  "draft_id": 123,
  "candidate_id": 1,
  "candidate_name": "Alice Chen",
  "job_id": 5,
  "job_title": "Senior Backend Engineer",
  "company_name": "TechCorp",
  "fit_score": 82,
  "cover_note": "I am excited to apply for the Senior Backend Engineer role...",
  "answers": [
    {
      "question": "Are you authorized to work?",
      "answer": "Yes, I am a US citizen.",
      "provenance": "candidate_answer",
      "question_id": 1
    }
  ],
  "needs_review_flags": [
    {
      "question": "Years with our tech stack?",
      "reason": "No candidate answer found for category: tech_stack_specific",
      "question_id": 5
    }
  ],
  "status": "draft",
  "created_at": "2026-03-16T14:30:00Z"
}
```

Errors:
- **404**: Draft not found or not in draft status

SLO: **< 500ms** (database retrieval + response assembly)

---

**PATCH /api/v1/approvals/{draft_id}**

Update answers and cover note.

Request:
```json
{
  "answers": [
    {
      "question": "Are you authorized to work?",
      "answer": "Yes, I am authorized without sponsorship.",
      "provenance": "candidate_answer",
      "question_id": 1
    }
  ],
  "cover_note": "I am very excited about this opportunity and believe..."
}
```

Response:
```json
{
  "draft_id": 123,
  "candidate_id": 1,
  "job_id": 5,
  "answers_count": 3,
  "status": "draft",
  "updated_at": "2026-03-16T14:32:00Z"
}
```

Errors:
- **404**: Draft not found or not in draft status
- **422**: Update validation failed

SLO: **< 500ms**

---

**POST /api/v1/approvals/{draft_id}/approve**

Approve draft and create snapshot.

Request: Empty body (no parameters)

Response:
```json
{
  "draft_id": 123,
  "status": "approved",
  "approved_at": "2026-03-16T14:35:00Z",
  "snapshot_s3_key": "approval-snapshots/1/123_1710599700.json"
}
```

Behavior:
- Creates immutable snapshot
- Stores in S3 (if configured)
- Transitions draft.status → 'approved'
- Cannot be undone (draft locked)

Errors:
- **404**: Draft not found or not in draft status
- **422**: Approval failed (S3 error, permission issue, etc.)

SLO: **< 2 seconds** (includes S3 write)

---

**POST /api/v1/approvals/{draft_id}/reject**

Request revisions (rejection).

Request:
```json
{
  "draft_id": 123,
  "reason": "Please provide more detail on your experience with system design and distributed systems. The current answer is too brief."
}
```

Response:
Same as GET (returns updated draft ready for re-editing)

Behavior:
- Logs rejection reason
- Keeps draft in 'draft' status
- Allows user to continue editing

Errors:
- **404**: Draft not found
- **422**: Rejection logging failed

SLO: **< 500ms**

### State Machine

```
DRAFT (M1.5 output)
  ├─ [GET] → ApprovalScreenResponse (read-only view)
  ├─ [PATCH] → UpdateDraftResponse (edit answers/cover)
  ├─ [POST /approve] → ApproveResponse (transition to APPROVED)
  └─ [POST /reject] → ApprovalScreenResponse (stay in DRAFT)

APPROVED (M1.6 output, ready for M1.7)
  └─ (locked, cannot edit further)

SUBMITTED (M1.7+)
  └─ (application sent to ATS)

FAILED

Optional: REVERTED (Phase 2 - if user rejects a submission)
  └─ → DRAFT (editable again)
```

### State Transition Rules

| From | To | Method | Condition |
|------|----|----|-----------|
| DRAFT | DRAFT | PATCH | Always allowed |
| DRAFT | DRAFT | GET | Always allowed |
| DRAFT | DRAFT | POST /reject | Always allowed |
| DRAFT | APPROVED | POST /approve | No hard blockers |
| APPROVED | - | Any | Blocked (locked state) |
| SUBMITTED | - | Any | Blocked |
| FAILED | - | Any | Blocked |

### Concurrency Model

**Last-Write-Wins** (simple, works for MVP):
- Multiple users can edit same draft
- Last update overwrites previous
- No conflict resolution
- No optimistic locking

**Phase 2 Enhancement: CRDT or Vector Clocks**
- Track edit timestamps
- Detect conflicts
- Offer merge UI or conflict resolution

## Usage Examples

### Frontend Integration (Svelte)

```svelte
<script>
  import { onMount } from 'svelte';
  
  let draftId = 123;
  let approval = null;
  let loading = false;
  let edited = false;

  onMount(async () => {
    const res = await fetch(`/api/v1/approvals/${draftId}`);
    approval = await res.json();
  });

  async function onApprove() {
    const res = await fetch(`/api/v1/approvals/${draftId}/approve`, {
      method: 'POST'
    });
    if (res.ok) {
      window.location.href = '/dashboard/applications';
    }
  }

  async function onReject() {
    const reason = prompt('Why request revisions?');
    if (reason) {
      const res = await fetch(`/api/v1/approvals/${draftId}/reject`, {
        method: 'POST',
        body: JSON.stringify({ draft_id: draftId, reason })
      });
      if (res.ok) {
        approval = await res.json();
        edited = false;
      }
    }
  }

  function onAnswerEdit(index, newAnswer) {
    approval.answers[index].answer = newAnswer;
    edited = true;
  }

  async function onSave() {
    const res = await fetch(`/api/v1/approvals/${draftId}`, {
      method: 'PATCH',
      body: JSON.stringify({
        answers: approval.answers,
        cover_note: approval.cover_note
      })
    });
    if (res.ok) {
      edited = false;
    }
  }
</script>

<div class="approval-screen">
  <h2>{approval.job_title} at {approval.company_name}</h2>
  <p>Fit Score: {approval.fit_score}/100</p>

  <div class="cover-note">
    <label>Cover Note:</label>
    <textarea bind:value={approval.cover_note} on:input={() => edited = true} />
  </div>

  <div class="answers">
    <h3>Screening Answers:</h3>
    {#each approval.answers as answer, i}
      <div class="qa">
        <p><strong>Q:</strong> {answer.question}</p>
        <textarea value={answer.answer} on:input={(e) => onAnswerEdit(i, e.target.value)} />
        <p class="source">From: {answer.provenance}</p>
      </div>
    {/each}
  </div>

  {#if approval.needs_review_flags.length > 0}
    <div class="review-flags">
      <h3>⚠️ Questions Flagged for Review:</h3>
      {#each approval.needs_review_flags as flag}
        <p>{flag.question}</p>
        <p class="reason">{flag.reason}</p>
      {/each}
    </div>
  {/if}

  <div class="actions">
    <button on:click={onSave} disabled={!edited}>Save Changes</button>
    <button on:click={onReject}>Request Revisions</button>
    <button on:click={onApprove} class="primary">Approve & Submit</button>
  </div>
</div>

<style>
  .approval-screen { max-width: 800px; margin: 0 auto; }
  .qa { border: 1px solid #ddd; padding: 12px; margin: 10px 0; }

  .reason { font-size: 0.9em; font-style: italic; }
</style>
```

## Performance

- **GET approval screen**: 50-150ms (1-2 DB queries)
- **PATCH update**: 100-200ms (1 update + commit)
- **POST approve**: 500-2000ms (includes S3 snapshot if enabled)
- **POST reject**: 100-200ms (metadata logging)

**SLOs:**
- All operations < 2 seconds
- S3 failures don't block approval (logged but non-blocking)

## Known Limitations (Phase 1)

- ❌ No conflict resolution for concurrent edits (last-write-wins)
- ❌ S3 snapshots optional (no hard requirement)
- ❌ No audit UI to view approval history
- ❌ No undo/rollback after approval
- ❌ No version history of edits

## Phase 2 Enhancements

- ✨ **Conflict Resolution**: CRDT or vector clocks for concurrent edits
- ✨ **Audit UI**: Show all edits, rejections, approvals with timestamps
- ✨ **Undo/Rollback**: Revert from 'approved' → 'draft' (one-time)
- ✨ **Activity Timeline**: Visual timeline of all actions on package
- ✨ **Mandatory S3**: All snapshots archived to S3 with signature verification
- ✨ **Multi-user Approval**: Two approvers required for sensitive jobs
- ✨ **Package Templates**: Save approved packages as templates for similar roles
