# M1.5: Application Package Generation (v1)

**Status:** ✅ Complete  
**Last Updated:** 2026-03-16  
**Commits:** [Implementation](#implementation-commits)

## Overview

M1.5 generates customized application packages by matching candidate profile & answer library to job postings. Each package includes:
- **Screening Question Answers**: Auto-matched from candidate library, resume, or profile
- **Cover Note**: Contextualized 3-5 sentence statement based on job + fit score
- **Needs Review Flags**: Questions that couldn't be auto-answered (manual review required)
- **Database Storage**: Persistent ApplicationDraft with JSON-encoded answers

**Typical Flow:**
1. User has completed M1.1 (candidate profile + answer library) and M1.3 (job discovery)
2. User generates package via `POST /api/v1/packages/generate` with `(candidate_id, job_id)`
3. System retrieves M1.4 fit score if available (for cover note context)
4. System generates answers (library match → resume fallback → profile fallback)
5. System flags unanswerable questions for human review
6. Package stored as ApplicationDraft with status='draft' (ready for M1.6 approval)

### Service Architecture

**File:** `backend/app/services/package_generator.py` (~380 lines)

**Class: PackageGenerationService**

#### Methods

**`generate_package(candidate_id, job_id, db) → dict`**
- Main entry point
- Retrieves candidate, job, fit score (M1.4) if available
- Orchestrates answer generation and cover note creation
- Stores as ApplicationDraft
- Returns dict with all package fields

**`_generate_answers(candidate, job, db) → (answers, needs_review_flags)`**
- Retrieves likely screening questions for job's ATS type
- For each question, matches candidate answer via `_match_candidate_answer()`
- Collects into answer list or flags as needing review

**`_infer_screening_questions(job, db) → list[AnswerLibraryQuestion]`**
- Strategy 1: Filter answer library by job's ATS type (e.g., Greenhouse)
- Strategy 2: Rank by `frequency_rank` (lower = more common)
- Returns top 5-8 questions

**`_match_candidate_answer(candidate, category, db) → dict | None`**
- 3-tier hierarchy:
  1. **Explicit Match**: Query CandidateAnswer table for this category
     - E.g., candidate answered "work_auth" → use that answer
  2. **Resume Fallback**: Extract from parsed resume data
     - E.g., extract years_of_experience, skills list
  3. **Profile Fallback**: Generate from profile fields
     - E.g., profile.remote_preference, profile.location

**`_extract_from_resume(parsed_data, category) → str | None`**
- Extract relevant fields from Resume.parsed_data (JSON from M1.1)
- Handles: work_auth, years_experience, technical_skills

**`_generate_from_profile(candidate, category) → str | None`**
- Generate default answers from profile fields
- Handles: work_auth ("I am authorized to work in the US")
- Handles: remote_preference (location + preference text)
- Handles: culture ("I value collaboration...")
- Handles: motivation ("I am excited to contribute...")

**`_generate_cover_note(candidate, job, fit_score) → str`**
- 4-sentence template:
  1. Opening: "I am excited to apply for the [role] at [company]"
  2. Fit-contextualized body:
     - if fit_score ≥ 75: "strong match"
     - if fit_score ≥ 50: "aligns well, eager to learn"
     - else: "interested and can add value"
  3. Location context: Remote/onsite alignment
  4. Closing: "opportunity to discuss / contribute"

### Answer Matching Strategy

The 3-tier hierarchy ensures answers are grounded in candidate data:

```
For each screening question:
  1. Check CandidateAnswer table (explicit answer by category)
     → BEST: User-provided or specifically saved answer
  2. Check Resume parsed_data (extracted by M1.1 parser)
     → GOOD: Factual resume data (skills, years, etc.)
  3. Check CandidateProfile fields (location, remote_preference)
     → OKAY: Profile-derived defaults
  4. None available
     → FLAG FOR REVIEW: Requires manual input
```

### Cover Note Generation

Cover note adapts based on fit score:

| Fit Score | Tone | Example |
|-----------|------|---------|
| 75-100 | Strong match | "My experience and skills are a strong match. I am confident I can make an immediate impact." |
| 50-74 | Aligned | "My background aligns well with many aspects. I am eager to learn and grow." |
| <50 | Interested | "I am interested and believe my skills can add value to your team." |

All notes include:
- Role + company name
- Location/remote context
- Closing statement

### Python Client Example

```python
import requests

response = requests.post(
    'http://localhost:8000/api/v1/packages/generate',

        'job_id': 5
```

### JavaScript/Frontend Example

```javascript
const response = await fetch('/api/v1/packages/generate', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    candidate_id: 1,

  })
});

const package = await response.json();

// Display answers
package.answers.forEach(ans => {
  console.log(`Q: ${ans.question}`);
  console.log(`A: ${ans.answer}`);
  console.log(`Source: ${ans.provenance}`);
});

// Display needs_review
package.needs_review_flags.forEach(flag => {
  console.warn(`⚠️ Review needed: ${flag.question}`);
  console.warn(`   Reason: ${flag.reason}`);
});
```

## Performance Characteristics

**Timing:**
- **First-time generation**: 200-500ms (includes all DB queries)
- **With M1.4 fit score available**: +50ms (additional query)
- **Cover note generation**: <10ms (template + string operations)
- **Database persistence**: 20-30ms

**SLO:** < 30 seconds (actual: 500ms typical)

**Database Queries:**
1. Retrieve candidate profile + resumes
2. Retrieve job posting
3. Query answer library (filtered by ATS type)
4. Query candidate answers (library matches)
5. Query fit score (M1.4)
6. Insert ApplicationDraft
7. Commit

**Scaling:**
- No N+1 queries (uses single query per entity type)
- JSONB storage supports future full-text search (PostgreSQL)
- ApplicationDraft unique constraint: (candidate_id, job_id) prevents duplicates

## Known Limitations & Phase 2 Enhancements

**Phase 1 Limitations:**
- ❌ No LLM-based answer synthesis (uses template matching only)
- ❌ English-only (hardcoded prompts + library in English)
- ❌ Limited resume parsing (depends on M1.1 parser output)
- ❌ No multi-language support
- ❌ No learning loop (answers don't improve based on feedback)

**Phase 2 Enhancements:**
- ✨ LLM-based answer generation ("Synthesize answer based on resume + job description")

## Implementation Notes

### Answer Matching Algorithm

```python
def match_answer(candidate, category):
    # Tier 1: Explicit library answer
    answer = query(CandidateAnswer)
      .filter(category == category)
      .first()
    if answer: return answer
    
    # Tier 2: Resume data
    if candidate.resumes[0].parsed_data:
      answer = extract_from_resume(parsed_data, category)
      if answer: return answer
    
    # Tier 3: Profile defaults
    answer = generate_from_profile(candidate, category)
    if answer: return answer
    
    # Tier 4: None available
    return None
```

### Cover Note Contextualization

```python
def generate_cover_note(candidate, job, fit_score):
    lines = []
    
    # Always include role + company
    lines.append(f"I am excited to apply for {job.title} at {job.company}")
    
    # Adapt tone based on fit
    if fit_score >= 75:
        lines.append("My experience is a strong match.")
    elif fit_score >= 50:
        lines.append("My background aligns well with this role.")
    else:
        lines.append("I am interested in this opportunity.")
    
    # Add location context
    if job.location and candidate.location:
        lines.append(f"Based in {candidate.location}, ...")
    
    # Close with call to action
    lines.append("I welcome discussing how I can contribute.")
    
    return " ".join(lines)
```
