## Overview

M1.8 provides a comprehensive dashboard for viewing all job applications submitted through the system. Users can see submission history, filter by status/company, view detailed results, and analyze submission trends.

**Key Responsibilities:**
- Display all submissions with current status
- Provide summary statistics (total, by status, completion rate)
- Support filtering and pagination
- Show submission run details and history
- Display per-company statistics
- Provide activity timeline visualization

---

## Acceptance Criteria

✅ **AC1:** Retrieve submission summary statistics  
- Count by status (draft, approved, submitted, failed)  
- Average and max fit scores  
- Submission completion rate  
- Time window filtering (days)

✅ **AC2:** List submissions with filtering  
- Filter by status, company name, date range  
- Pagination support (limit, offset)  
- Include run status if available  
- Sort by creation date (descending)

✅ **AC3:** Get detailed submission view  
- Full draft info (answers, cover note)  
- Associated job and candidate details  
- Complete run history (all attempts)  
- Error messages and results

✅ **AC4:** Company-level analytics  
- Aggregate by company name  
- Count, avg fit score, status distribution  
- Sorted by application count

✅ **AC5:** Activity timeline  
- Daily application count  
- Time window filtering  
- Chronological ordering

✅ **AC6:** API response format  
- Consistent error handling (404, 500)  
- Paginated responses with metadata  
- All timestamps in ISO format

✅ **AC7:** SLO Performance  
- Summary query < 500ms  
- List query < 1s  
- Detail query < 1s

## Usage Examples

### Svelte Frontend - Dashboard Component

```svelte
<script lang="ts">
  import { onMount } from 'svelte';
  
  let candidateId: number;
  let summary: any;
  let submissions: any[] = [];
  let loading = false;
  let selectedCompany: string | null = null;

  async function loadDashboard() {
    loading = true;
    
    // Load summary
    const summaryRes = await fetch(
      `/api/v1/dashboard/summary?candidate_id=${candidateId}`
    );
    summary = await summaryRes.json();
    
    // Load submissions
    const listRes = await fetch(
      `/api/v1/dashboard/submissions?candidate_id=${candidateId}&limit=20`
    );
    const result = await listRes.json();
    submissions = result.data;
    
    loading = false;
  }

  async function viewSubmissionDetail(draftId: number) {
    const res = await fetch(`/api/v1/dashboard/submissions/${draftId}`);
    const detail = await res.json();
    // Display detail modal with full history
    showDetailModal(detail);
  }

  async function filterByCompany(company: string) {
    selectedCompany = company;
    const listRes = await fetch(
      `/api/v1/dashboard/submissions?candidate_id=${candidateId}&company=${company}&limit=20`
    );
    const result = await listRes.json();
    submissions = result.data;
  }

  onMount(() => loadDashboard());
</script>

<div class="dashboard">
  <h1>Applications Dashboard</h1>
  
  <!-- Summary Stats -->
  {#if summary}
    <div class="summary-cards">
      <div class="card">
        <h3>Total Applications</h3>
        <p class="stat">{summary.total_drafts}</p>
      </div>
      <div class="card">
        <h3>Submitted</h3>
        <p class="stat">{summary.by_status.submitted}</p>
      </div>
      <div class="card">
        <h3>Completion Rate</h3>
        <p class="stat">{summary.submission_completion_rate.toFixed(1)}%</p>
      </div>
      <div class="card">
        <h3>Avg Fit Score</h3>
        <p class="stat">{summary.avg_fit_score.toFixed(1)}</p>
      </div>
    </div>
  {/if}

  <!-- Submissions List -->
  {#if !loading}
    <div class="submissions-list">
      {#each submissions as submission (submission.draft_id)}
        <div 
          class="submission-row"
          on:click={() => viewSubmissionDetail(submission.draft_id)}
        >

            {submission.draft_status}
          </div>
          <div class="score">{submission.fit_score}</div>
        </div>
      {/each}
    </div>
  {/if}
</div>
```

## Architecture

### Data Layers

```
API Route Layer
  ↓
DashboardService (Query Logic)
  ↓
SQLAlchemy ORM
  ↓
PostgreSQL Database
```

### Query Optimization

**Indexes (Recommended):**
```sql
CREATE INDEX idx_draft_candidate_created 
  ON application_drafts(candidate_id, created_at DESC);

CREATE INDEX idx_draft_status 
  ON application_drafts(status);

CREATE INDEX idx_run_draft 
  ON submission_runs(draft_id);

CREATE INDEX idx_job_company 
  ON job_postings(company_name);
```

### Service Methods

**DashboardService** provides 5 core query methods:

1. `get_submission_summary()` — Summary stats + time window filtering
2. `list_submissions()` — Paginated list with filters
3. `get_submission_detail()` — Full details + run history
4. `get_company_stats()` — Aggregated by company
5. `get_timeline()` — Daily activity counts

All methods work from a single database session and support proper error handling.

## Performance

### Expected Query Times

| Query | Typical Time | Worst Case |
|-------|--------------|-----------|
| Summary (30 days) | 150-200ms | 500ms (1000+ drafts) |
| List (20 results) | 200-300ms | 800ms (large dataset) |
| Detail | 100-150ms | 300ms (many runs) |
| Company Stats | 250-350ms | 1s (many companies) |
| Timeline (30 days) | 100-150ms | 500ms |

### Scaling Considerations

- Results scale linearly with candidate's submission count
- Multi-candidate queries will need aggregation logic (Phase 2)
- Dashboard UI should paginate at 20 results per page

## Limitations (Phase 1)

1. **Single candidate only** — Each endpoint operates on one candidate_id; Phase 2 could support user's candidate portfolio

2. **No export functionality** — Cannot download submission history as CSV/PDF; Phase 2 could add export

3. **No real-time updates** — Dashboard is static per query; Phase 2 could use WebSocket for live updates

4. **Basic filtering** — Only status and company name; Phase 2 could add date range, fit score range, ATS type

5. **No drill-down analytics** — Cannot see detailed trend analysis; Phase 2 could add charts (conversion rate by company, etc.)

6. **No search** — Cannot search by job title or description; Phase 2 could add full-text search

7. **No applications synced from platforms** — Only shows drafts/submissions from this system; Phase 3 could sync accepted/rejected status from portals

## Glossary

**Draft Status:** draft (unsaved) | approved (ready to submit) | submitted (submitted successfully) | failed (submission error)

**Run Status:** running | completed (success) | failed (error) | cancelled (user cancelled)

**Fit Score:** 0-100 score indicating job-candidate compatibility

**ATS Type:** greenhouse, lever, linkedin, workday, etc.

**Completion Rate:** Percentage of drafts that reached "submitted" status
